"Our system" — what it actually is, and where it lives

Fair question — it never had a name or an installer, so it feels like "just habits." Half right. There is a real, convention-driven planning layer; it's just unbundled across files, and the only thing enforcing it is habit plus the CLAUDE.md prompt. Here are its four layers and exactly where each one sits on disk.

So "vibe coding" applies to one layer — the enforcement — not the whole thing. The artifacts and rules are concrete files you can open; what's missing is the machinery that checks them. That single missing layer is precisely what OpenSpec supplies.

Not part of the system, despite the name: custom prompts / prompt-stash is a harness convenience for re-running saved prompts — it doesn't hold any planning convention.

This is also what "dual-write" meant: during migration, each new feature writes both sets — the old artifacts above and the OpenSpec changes/<id>/ — until Phase 3/4, so nothing goes dark mid-switch.

Console — drive OpenSpec for real

You know the old system cold — plan.md, plans/*.md, the ship ritual — but may be new to OpenSpec. So every row starts from what you did before, then runs the OpenSpec equivalent for real (in the repo root, through this app's server) and prints the output in the log below. Rows tagged New to OpenSpec had no old equivalent — they're the unlocks.

Honest scope: this makes the workflow one-tap and auditable — it does not make the port "under control." A button you may click is a convenience, not a gate. The thing that actually enforces the spec — validate --strict blocking bad work — has to live in CI / a git hook, outside this app. The console shows you the state; CI is what makes it mandatory.
