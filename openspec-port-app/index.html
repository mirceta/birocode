The lifecycle we're adopting

  1. propose a change folder
  2. specify deltas + scenarios
  3. design the approach
  4. implement from tasks.md
  5. archive → folds the delta into the living baseline

The fold on archive is the prize — the one thing our forward-looking plans lack: a queryable "what does the system do today?" Only one OpenSpec artifact is net-new to us: specs/<cap>/spec.md. The rest maps onto things we already write.

How OpenSpec works — the mechanics

Beneath the port project: how the tool itself is built. OpenSpec keeps what's true now strictly apart from what we propose to change, and ships only by folding one into the other. The Cockpit tab is a live, read-only window onto exactly the files described here.

Mental model: the baseline is the current truth, a change is a diff against that truth, and the archive is applied diffs. Shipping is literally baseline + delta → new baseline.

Two worlds, plus an archive

📚 The baseline
openspec/specs/<cap>/

"What the system does today." One folder per capability (chat, files, openspec-cockpit…), each a list of Requirements + Scenarios. Read it with openspec spec list.

🛠️ In-flight changes
openspec/changes/<id>/

A proposal not yet true: intent, design, a task checklist, and delta specs that list requirements to ADD / MODIFY / REMOVE. Listed by openspec list.

📦 The archive
openspec/changes/archive/

Shipped changes, date-stamped (2026-06-22-<id>). On archive the delta is folded into the baseline and the change retires here — the record of how the baseline got its shape.

Anatomy of one change

The four files inside a changes/<id>/ folder — each answers a different question.

proposal.mdwhy + what

The pitch: Why, What Changes, Impact (which specs + code it touches). Created by /opsx propose.

specs/<cap>/spec.mdthe delta

The contract the code must meet: requirements to ADD / MODIFY / REMOVE, each with WHEN/THEN scenarios. The only net-new artifact OpenSpec adds.

tasks.mdthe checklist

[ ]/[x] items. The Cockpit reads these boxes for its "15 of 19"-style completion — openspec show --json omits them, so it reads the file directly.

design.mdhow

The technical approach — component shape, state flow, trade-offs. Required for non-trivial changes; a trivial one may skip it.

The spec grammar

Every requirement has the same shape in both the baseline and the deltas — that regularity is what lets the CLI (and the Cockpit) parse it.

# Capability heading (e.g. "Chat")

### Requirement: <imperative, testable statement>
The system SHALL …            # normative "must"; SHALL NOT for prohibitions

#### Scenario: <a concrete case>
- WHEN <trigger>            # the situation
- THEN <observable result>   # what must be true after

The delta fold — what archive does

A change never edits the baseline directly. Its delta lists operations; openspec archive <id> replays them onto specs/<cap>/ and moves the change to the dated archive.

ADDED a new requirement appears in the baseline
MODIFIED an existing requirement is replaced in place
REMOVED a requirement is deleted from the baseline

openspec validate --strict gates the delta's shape before any code is written; archive applies the ops and stamps the change into changes/archive/. After that, the baseline reflects the new truth and openspec list no longer shows the change as active.

Workflows — the five jobs you'd actually do

OpenSpec is two modes wearing one CLI. You read the living truth — every read is a command, so the Console runs it all. You write a change — but the thinking is authored by hand; only the bookends (scaffold, gate, ship) are commands. Tap a workflow for its steps.

✓ runs in Console ✍ authored by hand ⌨ terminal only

The read workflows are pure commands — the Console covers them completely. The change loop is where the 4-ported / 7-by-hand split on the Adopt tab comes from: its bookends are buttons, its middle you write yourself.

Console — drive OpenSpec for real

The executable twin of Workflows — the same five jobs, in the same order, but here each command is a real button (run in the repo root, through this app's server, output in the log below). The ✓ steps are buttons; the ✍ steps are authored by hand — fill the small field beside one (e.g. change-name, typed once and shared across the steps) and ⧉ Copy prompt puts a fully filled-in instruction on your clipboard to hand straight to the agent; ⌨ is terminal-only. Every step also notes how you did it in the old system (was); Adopt has the fuller side-by-side.

Honest scope: this makes the workflow one-tap and auditable — it does not make the port "under control." A button you may click is a convenience, not a gate. The thing that actually enforces the spec — validate --strict blocking bad work — has to live in CI / a git hook, outside this app. The console shows you the state; CI is what makes it mandatory.

The server runs a fixed whitelist — you pick a step, it builds the command — so nothing here can run an arbitrary shell. The WRITE groups change files in the repo; the READ groups don't.
