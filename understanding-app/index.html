"Only one person" — why a personal questionnaire is the wrong lock

A "questions only they'd know" gate is knowledge-based auth (KBA). Pick a method and watch how far into your life it leaks — i.e. who else could pass. Red rings can get in; the green centre is the one person you actually want.

Guess-resistance

The reframe: a questionnaire is just a low-entropy, non-rotatable, semi-public password — entropy: lowrotatable: nosecret: partly public — and a real passphrase or passkey beats it on every one of those.
What gives you "only this one person", properly: ① give that person the strong password (B) — it already is info only they know, but high-entropy & rotatable · ② enrol a passkey on their device (D) — possession + Face ID, can't be shared by knowing facts · ③ a unique secret invite link. Keep the trivia as fun flavour behind the lock, never as the lock.

The gap, and how to close it: B = password + cookie = two bearer secrets, so it is breached if either one leaks — that caps it at High. Swap B's password factor for a passkey (D) and you keep the cookie's convenience but gain both properties (origin-bound, non-exportable): the "phishing" and "stolen credential" rows flip red→green and B effectively rises to Highest. That's why D is framed as B's natural follow-up, not a rival.

Two jobs: invisibility vs. the lock

Calling both of them "gates" hides the point — they do different jobs. The IP gate makes the app invisible; the password is the lock. One isn't better than the other; they cover each other's blind spots.

🌐 The whole internet — hostile by default
🕶️ IP gate — invisibility. Unapproved addresses get a 403 at the door; your app code never runs. Stops scanners, bots, and bugs in routes you forgot.
🔒 Password / passkey — the lock. Proves who you are. Stops anyone who reaches the door but doesn't hold the secret.
🧠 Your app

The lock is only reachable by someone the invisibility layer already let near the building — and the invisibility layer can't tell who anyone is. Different jobs.

The trap you hit: rows 4 and 6 are the IP gate being asked to authenticate — to judge who — which it's bad at, because addresses are shared (CGNAT) and reassigned (4G). The fix isn't to trust it more or to delete it; it's to move the identity job onto the device cookie, and leave the IP gate doing the one thing it's genuinely great at: keeping the app invisible to everyone you haven't approved.

Who actually attacks you — automated vs. targeted

Two very different attackers. Pick one and walk it through each model — watch which barriers it must climb and where it gets stopped. The key fact: they don't happen equally often.

The asymmetry that settles it: the attacker that's common (the bot) is untargeted — and it's exactly what invisibility stops, so B makes you invisible to it. The attacker that's scary (cookie + password) is rare and targeted — and it's the bar B imposes, not soft. Going soft trades away your defence against the common one and lowers the bar for the rare one.

The solution — strict gate + trusted-device cookie (Approach B)

In one line: keep the strict IP gate exactly as it is, and mint a long-lived trusted-device cookie the first time you approve someone. The gate then admits an approved IP OR a valid cookie — so a friend's rotating 4G IP never re-bars them — while everyone you haven't approved still hits the same hard 403. The password stays as the second factor.

The rule the gate follows

✓ Approved IP

Pass — unchanged from today.

✓ Unapproved IP + valid cookie

Pass — the 4G-rescue case. The device is remembered, the address isn't. (Optionally records the new IP, tagged via device cookie.)

✗ Anything else

Hard 403 + rejection page, byte-for-byte as today. You approve manually. No fall-through, ever.

🍪 The trusted-device cookie

HttpOnlySecure (HTTPS) SameSite256-bit random ~180-day slidingonly its hash stored server-side revocable

Minted only on a successful login from an already-approved IP — so a 403'd stranger can never obtain one.

🔧 What gets built

① DeviceTokenService — issue / validate-and-slide / revoke, hash stored beside sessions · ② mint the cookie on login success · ③ IP gate checks the cookie before its 403 · ④ a "Trusted devices" list with Revoke in the desktop GUI · ⑤ DeviceCookieDays config. (See the change's tasks.md.)

Four things we must get right

1Revocation is mandatory. Removing an IP no longer evicts a cookie-holder — so the "Trusted devices" list must let you revoke, and removing a guest offers to revoke their tokens. Without this you can't fully evict anyone.

2The cookie proves the device, not the person. New phone / cleared cookies → one more approval. Intended — it's what stops a lost phone being a permanent backdoor.

3Long, sliding lifetime (~180 days) so the call-you event is rare; a quiet friend re-approves once.

4HttpOnly + Secure + SameSite — the cookie now skips the IP gate, so it must be unreadable to JS and HTTPS-only.

What you get

✓You personally approve every first entry — the human allowlist stays.

✓Everyone you haven't approved still hits a hard 403 — the app stays invisible to the automated internet.

✓An approved friend is never re-barred when their 4G IP changes — no more phone calls.

✓The password remains a second factor behind the gate — a leaked cookie alone doesn't grant access.

✓You can fully evict anyone by revoking their device + IP.

~Residual risk (honest): a stolen, un-revoked, HttpOnly cookie + the password — a targeted attack only, the reassuringly hard chain from tab ⑥. Closeable later by swapping the password factor for a passkey (tab ④).

The whole model — what the proposal & design actually change

The change has two halves: (1) remember the device (trusted-device cookie, tabs ①–⑦) and (2) delete the in-app permission layer so the two gates are the entire authorization model. Here's the shape of both.

Before → after: the authorization stack

Before

🕶️ Gate 1 — strict IP allowlist
🔒 Gate 2 — password / session
🎚️ Per-project permission preset — Read-only default, Standard denylist (no rm -rf, curl…)
💻 Runs as the OS account

After

🕶️ Gate 1 — strict IP allowlist OR trusted-device cookie
🔒 Gate 2 — password / session
🎚️ Per-project permission preset — REMOVED
💻 Full access — bounded only by the OS account

Two edits: Gate 1 gains the cookie (so a rotated IP isn't a re-bar), and the whole permission tier is deleted (so there's no second gate on what a passed user does).

Removing the permission system — what goes, what stays

🗑️ Removed

✗The presets (Read-only / Edit-only / Standard / Full)

✗ApplyPermissionFlags + the Standard denylist — chat runs unrestricted

✗The Read-only safe default for unconfigured projects

✗Desktop preset picker + the read-only web badge

✗The project-permissions spec capability

✅ Kept (not per-user authorization)

✓The IP + password gates — the auth model itself

✓X-Repo-Id — repo routing, not access control

✓Repo visibility (basic/advanced) — a UI filter only

✓AutopilotGate — a host-only feature on/off switch

✓The user-selectable read-only "ask" mode — a chosen tool

The boundary becomes the OS account — so size it

With the permission tier gone, the only thing limiting a passed user (or a breached cookie + password) is the Windows account the harness process runs as — not a per-user login; the End User never authenticates to Windows. Pick a scenario and see how far a breach reaches:

The thesis of the design: for a single-Operator harness with a few hand-approved devices, two overlapping trust tiers are redundant — one clear boundary (the gates) beats two. The price is full-access-by-default, so the design's paired recommendation is to run the harness under a dedicated least-privilege OS account: that's what turns "bounded by the OS account" into a sandbox you chose, and keeps a worst-case breach contained.

Action audit — trust but verify

The counterweight to removing permissions (tab ⑧). We no longer restrict a gate-passed user — so instead we record what they do. Every prompt and every mutating action is logged, attributed to who, append-only — for accountability now and forensics if a device or password is ever compromised.
