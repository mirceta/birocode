The 🧠 button that vanished — a version-skew diagnosis

The Ask for understanding button disappeared from live. It looked like an auto-rollback — it wasn't. The backend had the feature; the frontend bundle being served was a day older and predated it. This walks the diagnosis, the root cause, and the no-restart fix.

Symptom: a feature that shipped is suddenly gone

The 🧠 dock button was committed, deployed, and verified — then it wasn't on the live UI. Two explanations fit the symptom. Click each to run it against the evidence.

Hypothesis A — the auto-rollback fired

swap.ps1 arms a 15-min dead-man's switch. If nobody ran keep.ps1, live could have reverted to the last-good build, taking the button with it.

Hypothesis B — backend/frontend version skew

The deploy refreshed the backend but left a stale frontend bundle in the run dir. The route exists; the served JS just doesn't know about the button.

A rollback would revert everything to 6/28 — backend included. So the deciding question is: does the live backend still know the new route? Run the test.

The served bundle (6/28) is the only one missing the token — and it's also the oldest. A fresh build with the button exists in client\dist, but it was never copied into the run dir. That gap is the whole bug.

Why a fresh backend still renders an old button-less UI

Kestrel serves the static files in run-bin\client\dist per request — independently of the running C# process. So a stale dist silently downgrades the UI while the backend is brand new. Send a request and watch.

The backend never enters this path

The C# controller answers /api/understanding/*. But the shell that decides whether to draw the button is plain static JS. New backend + old shell = no button.

Why "didn't show up" is a known trap

In self-dev, the frontend lives as static files, not as part of the process. So a backend-only deploy leaves dist stale — the recurring "frontend change didn't show up" gotcha.

If the rollback had fired, the live backend would have reverted to 6/28 and the route would 404. It answers. So the backend is the new build — only the served JS is old. That's skew, not rollback.

The fix: mirror the fresh dist into the run dir — no restart

No rebuild needed: the correct bundle already existed in client\dist. Just mirror it into the run dir with robocopy /MIR. Because Kestrel reads static files per-request, the next reload serves the new bundle — no process restart. Press Run fix.

Two caveats handed back to you

① Hard-reload your browser

The old index-BKUDIuJp.js may be cached client-side. Reload to pull the new hashed bundle.

② Switch to Advanced mode

The button is gated to Advanced UI mode (understandingAgent: 'advanced'). In Simple mode it stays hidden even with the right bundle.
