Why was my source refused?

The harness event feed's collector now tells you which gate refused a watched harness — instead of always saying "needs a credential".

The problem this fixes. A watched harness refuses a pull for two very different reasons: its IP gate didn't let us through (no credential will ever fix that — the remote operator must approve our IP), or the credential was missing/wrong. Both used to surface as one status, "needs-credential" — sending you down the wrong fix.

All four refusals are amber "alive" states — the host answered; it just said no. Only a host that never answers goes red. The distinction applies from the very first probe after "Add harness", so the add form already names the actual problem. The 403's JSON body ({ error, ip }) is parsed best-effort, for the detail line only — a bare 403 from an older or foreign gate still classifies as ip-blocked from the status code alone.
