From one browser-polled feed → a backend fleet collector

An interactive walkthrough of the proposal. The harness already publishes its own events read-only (GET /api/events, seeded with turn.ended). This change moves the listening onto the backend and lets it aggregate many harnesses. Everything below is a simulation — no real network calls.

Two gaps in today's pilot

Gap 1 · listening lives in the browser

The events-app holds the "started" flag and the watermark in client JS. Reload the page and the feed stops — you must press start again. Nothing listens when no tab is open.

reload = stop & restart

Gap 2 · one harness only

It can watch only the harness serving it. The original proposal pointed at a collector that aggregates many harnesses — that's the goal here.

can't watch a fleet
The fix for both is the same idea: make the backend the source of truth for "what am I listening to, and how far have I read," and let the frontend be a thin observer.

The model: sources → backend poller → one merged feed

A backend CollectorService owns a list of sources. A background loop pulls each active source and merges everything into one stream, tagged by source. Press the button to run one poll pass.

The guarantees (and your four decisions)

👁️
Read-only toward every watched harness

The collector only ever issues GET /api/events to a source. It never causes or exposes an action on a watched harness. The only writes are to its own source list — the same category as registering a repo.

🔒
Credentials are write-only & encrypted at rest

A remote's credential is entered once, stored encrypted (Data Protection / DPAPI-backed keys), never returned in any response, and scrubbed from logs. your call #1

🌐
Any address is allowed

No allowlist — operator-entered, behind auth. Blast radius is bounded by short timeouts and per-source isolation. your call #2

🪞
Self is a built-in source

This harness is always a source, read in-process and active by default — "surely accessible" — so the local feed is always present without pressing start. your call #3

💾
Source list persists; events stay in memory

The sources survive a harness restart (listening auto-resumes); the aggregated events are an in-memory ring, like the feed. your call #4

The host cue: a sound on the machine, not the browser

Independent of any open tab, the computer running the harness can make a noise each time the collector ingests a new event — so an operator across the room knows something landed. It's an operator toggle (POST /api/collector/sound), debounced, played on a background thread, and best-effort: a host with no audio just stays silent.

Two modes — beep (default) vs voice

Beep plays the Windows notification tone (falling back to Console.Beep). Voice instead speaks, naming the source that finished — "agent {label} has finished" — through the default audio device via Windows SAPI (SpVoice over COM, no new dependency), in a soft, soothing female voice. Because the phrase says the label, every source now requires a label when you add it.

The selected mode is persisted (survives a restart) and the Test button plays whichever mode is live. If the voice path is ever unavailable, it falls back to the beep, so the cue is always best-effort.

host is quiet
This demo uses your browser's speech/beep to stand in for the host. In production the sound comes out of the host machine's speakers on the server side — no browser needed.
