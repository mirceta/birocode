📡 The Traffic Monitor

How the harness now measures its own HTTP throughput — and tells you when it's getting high. (OpenSpec change: traffic-monitor, branch feat/traffic-monitor)

Every open tab is a metronome

The web UI keeps itself live by polling: the Dashboard snapshot, account chips, event console, files viewer, understanding panel, host clock … most tick every 5 seconds. One phone tab is cheap. But the volume multiplies with every open tab, every summoned panel, every device — and until now nothing measured it. The server only had a manual request counter; nobody counted bytes.

Below: each row is a poller; a pulse is one request. Open tabs and watch requests/second climb.

One middleware, wrapped around everything

The counter is a Kestrel middleware registered outermost — before even the IP filter. So the numbers are true wire volume: static assets, the localview proxy legs, even rejected requests. Nothing opts in; nothing can forget to count.

Streamed responses (the chat's SSE stream — precisely the heavy ones) have no Content-Length, so the middleware swaps Response.Body for a pass-through CountingStream that tallies every write as it flows. No buffering — streaming behavior is untouched (verified with a real CLI turn through the isolated instance).

request travelling in response bytes counted on the way out

Attribution collapses IDs so the table stays readable: /api/repos/A/events and /api/repos/B/events are one bucket — GET api/repos/{repoId}/events. Non-endpoint traffic falls back to a normalized prefix (assets/*, api/localview/*), capped at 100 buckets with an other overflow.

900 one-second slots, indexed by clock modulo

Per bucket, counts land in a ring of 900 slots (15 minutes), slot = unixSecond % 900. A slot is lazily reset when its stored second no longer matches — so idle buckets cost nothing and rates decay to zero on their own. No database, no persistence: restart the harness, counters reset. This is monitoring, not billing.

Watch the write head sweep the ring (sped up; showing 60 of the 900 slots). Reads answer "rate now" by summing the trailing window — 10s for the live number, 60s for the average the high decision uses.

write head (current second) trailing 10s — "now" trailing 60s — feeds the high signal

The Traffic panel — and the amber line

Summon 📡 Traffic from the Dashboard's panel rail (Advanced mode). The header shows req/s · KB/s even collapsed; the body has a 60-second sparkline, the threshold line, and the top talkers. GET /api/traffic serves it all in one small JSON.

"High" is decided server-side: when the 60s average of response bytes/s exceeds TrafficHighBytesPerSec (appsettings, default 512 KB/s) the report says high: true, and the panel + rail chip turn amber. One definition, every consumer agrees.

threshold: 512 KB/s

The panel's own 5-second poll of /api/traffic is counted too and shows in its own table — deliberate honesty, not a bug.
