Why fixed categories, not user groups?

The first draft of this change planned user-defined groups with manual reordering. Superseded: the custom library was a temporary discovery vehicle — now that the practiced set is known, it belongs in version control like any other harness constant. Categories are assistant-identified and shipped with the code; reordering means editing promptCatalog.js in a commit, not dragging rows on a phone.

The hiding rule (why nothing shows twice)

A custom whose normalized text (trim, collapse whitespace, lowercase) matches any catalog text — base wording, .legacy wording, or a retired alias — is a promoted copy and is hidden from the pop-up. The store keeps it on purpose:

  • Autopilot builds its routine label space from _prompts.List() — deleting would silently shrink the recommender's vocabulary.
  • Rollback to an older build just shows the flat list again; no data event either way.
  • If a hidden copy's text is later edited (e.g. in the Routine-prompts tab), it stops matching and honestly reappears under New ideas as a new idea.
Promoted verbatim. Some promoted texts still speak the old plans-era language ("table in our plan"; "merge to master" survives only behind the Old-system toggle). Modernizing wording is a separate, deliberate edit — the catalog never rewrites silently.
