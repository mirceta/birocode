Maximize chat to fill the dock

An agent dock (the "phone" tile on the dashboard) stacks several non-chat chrome regions above the chat — they eat ~40% of the tile's height. This change adds one toggle, next to Tool Calls in the chat toolbar, that collapses that chrome so the chat fills the dock. Click the same button again to restore. The problem & the rule

Top-to-bottom, .phone is a flex column: fixed-height chrome regions, then a flex-filling .phone__screen that holds the embedded <Chat>. The chrome is useful but always-on — there was no way to reclaim its space to focus on the conversation.

  • One button, two states. The same toggle maximizes and restores — pressed/active when maximized, with an aria-label.
  • Collapse, don't unmount. A phone--chat-max modifier sets the chrome to display:none; the <Chat> subtree stays mounted, so scroll position, chat state, and the live stream survive the toggle.
  • The composer never hides. It lives inside .phone__screen, not the chrome — so it (and the toolbar) stay usable while maximized.

Key design decisions

State owner
PinnedAgent, not Chat. It renders the .phone__* chrome, so the chatMaximized boolean + toggle live there and pass down into the embedded chat.
Collapse via
CSS modifier class, not a remount. Single source of truth in CSS; mounted-but-hidden chrome is cheap and avoids layout churn / lost chat state.
Button home
In .chat__bar, beside .chat__tools. Mirrors the Tool Calls button's pattern: aria-pressed + an active class + i18n title.
Gate
Advanced mode only. The dock and Tool Calls button are already Advanced-only — no extra gate needed.

The one rule that explains the whole thing:
  • The chrome (.phone__bar/lanes/apps/git/discover) is what collapses; the chat in .phone__screen is what grows.
  • State is per-dock (one tile maximizing leaves the others alone) and ephemeral (resets to normal on reload — nothing is persisted, no API).
  • It's a pure additive frontend change: revert the commit to roll back, nothing to migrate.
