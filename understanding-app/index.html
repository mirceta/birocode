Discovery that survives a refresh — and now runs what it finds

"Discover local apps" is a per-repo, read-only agent scan fired from a dock's button. The first change (discover-local-apps-resilient) made it backend-owned: the scan runs to completion regardless of the browser, the dock reattaches on load, and each call gets its own metadata. Demo A & B below show that. The follow-up (discover-local-apps-run-controls, Demo C) closes the loop: the scan also extracts each app's start command, the harness reports a live running flag (computed from the port, never frozen into the scan), and the dock gains Run and Check buttons so you can start a discovered app and confirm it came up without leaving the dock.

What happens on refresh — After

The core fix

  • Backend owns the run. A singleton LocalAppDiscoveryJobs holds one DiscoveryJob per repo — status (running|done|error), result, timestamps — and the scan runs on a background task with the job's own cancellation token, never HttpContext.RequestAborted.
  • Start-or-join. GET /api/local-apps/discover registers or joins the repo's job and returns the current state immediately; it no longer blocks the run on the request.
  • Reattach. GET /api/local-apps/discover/status lets the dock, on mount, rejoin a running scan (spinner) or pick up a result/error that landed while the page was away — driven by the existing ~5s dock poll.

The fix — per-call identity (no gateway change)

Give every call a unique app name — StructuredAskRunner sends claudeweb-structured-ask#<callId> (a fresh GUID per call) instead of the shared base name. The gateway already resolves a call's record with FindLatestRecord(app); because the name is now unique to this call, "latest record for this app" is unambiguously this call's own record. The cross-wiring is fixed entirely in birocode — zero edits to the shared ClaudeMonitor gateway or client. Trade-off: the gateway's stats/dashboard gains one #<id> entry per discovery call.

Why running is computed live, not scanned

A scan takes ~30–60s; whether an app is up changes second to second. A running flag baked into the scan result would be stale the instant the scan finished. So the harness computes running at fetch time from the port itself (IPGlobalProperties.GetActiveTcpListeners() — in-process, no shell, no network). Check is simply "re-fetch status," which recomputes every row's dot — so it's always accurate, and it even reflects apps started outside the harness.

Run uses the scanned command, resolved by port

POST /api/local-apps/run { port } looks up that port's finding in the repo's latest scan and launches its stored startCommand detached in the app's folder. The client only sends a port — the command executed is the one discovery extracted, never a string off the wire. After a Run the dock waits a beat, then auto-Checks so the dot flips green once the server is listening.

Stop?
v1 gap none. No PID tracking yet; liveness is read off the port, so there's nothing retained to kill. A Stop endpoint is a separate, larger change.

Key design decisions

Scope held fixed

  • Payload is additive: { name, port, folder, evidence } plus the new optional startCommand and a harness-computed running — old callers unaffected.
  • Read-only scan policy — unchanged (Run is a separate, operator-initiated endpoint).
  • Single-repo-per-call scoping — unchanged.
  • GET /api/local-apps/discover stays backward-compatible (completed shape unchanged).

Decisions & trade-offs

Reuse RunSession?
No — borrow the idea, not the machinery. RunSessionService is built for seq-numbered streaming chat turns; discovery is one-shot with a typed JSON result. A smaller purpose-built store fits.
Persist state?
No — in-memory, one job per repo. A harness restart just means "no recent discovery." Latest-only overwrite avoids unbounded job leak.
Streaming?
No — only running/done/error. Simple interval polling for parity with the existing dock poll; no held connections.
User cancel?
v1 gap none. We removed the only cancellation source. A stuck run relies on the CLI runner's existing timeout; a cancel endpoint is out of scope, addable later.
The one rule that explains the whole thing:
  • The browser no longer owns the scan — the backend does. The request can come and go (refresh, disconnect); the job and its result live server-side, so the agent call is never wasted and the dock just re-derives its view on load.
  • Each call gets its own identity at the gateway, so concurrent discoveries can't swap metadata.
  • Everything else (payload, read-only policy, per-repo scoping, the route's completed shape) is held fixed — it's an additive change; rollback is a straight revert with no persisted state to undo.
