“It deployed fine, but my browser showed the old version.”

Why this kept happening — and the three things that now stop it.

The trap: a cacheable index.html

The build splits into a tiny index.html shell and big content-hashed JS files (index-BLqz5rzZ.js). The hash is the version. index.html is the one file that points at the current hashes — so if the browser (or the off-box IIS+ARR proxy) caches index.html, it keeps loading yesterday's hashes. The new code sits on the server, untouched.

index-OLD.js

⚠ Stale — the cached shell pins the OLD hash.

The fix — three layers

  1. 1 · Server: no-store on the shell. index.html & version.json now say “never cache me”; the hashed /assets/* stay cached forever (the name changes when the content does). So a reload always fetches the fresh shell → fresh hashes. root cause
  2. 2 · Force refresh button (Settings ▸ Maintenance). On demand: wipes Cache Storage, unregisters service workers, and reloads with a cache-bust — a guaranteed clean slate, even through a stubborn proxy. the button you asked for
  3. 3 · The auto-banner now bites. The existing “new version available” banner used a plain reload (which could re-serve the cached shell). It now runs the same thorough clear. closes the leak
