The frame host layer what the user never sees

One fixed layer at the app root owns every iframe. Pages only render an empty slot; the host projects the live frame over it. Leave the tab → the frame is just display:none, never destroyed.

Slot projection, running live in this page

The mechanism that shipped: the page renders an empty slot div where the app should appear; the iframe lives in a root-level host and is projected over the slot's rectangle (ResizeObserver + capture-phase scroll listeners, batched into one requestAnimationFrame). Lifetime rules that shipped

Navigationonly unregisters the slot → frame hides, state lives.
Repo switchreleases the Local tab's frames (verified: 2 → 0).
Dock removed / app deletedreleases that dock's / that app's frames.
LRU cap 67th frame evicts the least-recently-visible — but only hidden, non-pinned frames; a visible frame is never yanked.

Two behaviors the browser run made visible

Neither is a bug — both are the spec doing its job, and both are worth knowing before you test.

1 · Same app, two surfaces → two independent frames

Midway through the dock stage, the selector iframe[src*="statea"] suddenly matched two elements: the Local tab's kept-alive frame and the dock's new one. That's per-surface frame identity working exactly as specced — each surface gets its own document, own state, own ↻.

Those are the actual values from the run — the two documents never bled into each other, and refreshing one left the other untouched.

2 · Fully scrolled-out slot → the frame hides itself

The dock test first "failed": the phone's screen sat at y = 982 in a 950-px-tall viewport, so the host — correctly — hid the frame whose slot had zero visible area. The fix was in the test, not the code: scroll the slot into view like a real user, and the frame appears and tracks.

You already have this demo live: go to view 2 and scroll the slot fully out of the grey page — the status readout flips to hidden (slot off-screen), exactly what Playwright saw.

Why it matters: frames also reposition live while scrolling — the same capture-phase listener that hides an off-screen frame keeps an on-screen one glued to its slot.
