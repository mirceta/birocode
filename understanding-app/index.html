Five loose sound buttons are now one Sounds tab

OpenSpec change promote-sound-settings-to-tab is implemented and verified. The event feed's five sound controls used to sit crammed in the page header — a Device toggle, a 🎵 Sounds button that opened a floating modal, and three host-cue buttons — with no grouping, so nobody could tell what they were or how they related. This change promotes them into a fourth first-class tab (Sounds), laid out as a settings dashboard. The trick: move the markup, keep every element ID, so all the playback / IndexedDB / host-cue logic keeps working with zero rewiring.

Notice what didn't move: the 🖥 Display mode link stays in the header. Only the sound controls migrate — the two "unknown" buttons turn out to be the two poles of where a sound plays: this device vs. the host PC.

Why "move markup, keep IDs" is the whole point

The controls bind by id

Every handler already looked its element up by ID — #snd, #hsnd, #hmode, #htest, and the per-type grid rows. Relocating the DOM node doesn't touch the handler, so renderSnd(), playCue(), the host fetches and the IndexedDB assign/clear all keep working untouched.

The modal was pure chrome

The 🎵 Sounds button and floating overlay existed only to open and close the custom-audio editor. In a dedicated tab that editor is always on screen, so the opener, the #sndClose ✕, the backdrop and the three open/close handlers are gone — the grid renders inline.

The tab system was already there

A generic CSS rule shows the panel whose data-tab matches body[data-tab], and the resolver routes any tab name. Adding sounds meant one tab button, one panel, one CSS selector, and one array entry — no new show/hide mechanism.

Device vs. Host — the real fix

The confusing pair wasn't redundant, it was two scopes: Device = beep on the phone/browser you're holding (localStorage, private); Host = beep on the machine running the harness (shared server state, fires with no browser open). Grouping + one-line explainers make that legible.

How a Sounds selection travels

  1. Pick the tab — click Sounds, or deep-link ?tab=sounds. The URL query wins, else localStorage, else Activity.
  2. It's remembered — the choice is written to localStorage["eventsapp.tab"], so a reload with no query re-opens Sounds.
  3. CSS reveals the panel — body[data-tab="sounds"] flips #panel-sounds visible; the other three panels hide.
  4. Display mode is different — ?display=1 is tabless and force-shows every panel stacked, Sounds included, mirroring how Agents & GitHub already appear.
Verified headlessly (Playwright): the tab appears with no loose buttons left in the header, ?tab=sounds selects it and survives a reload, the Device toggle flips and persists, a custom file assigns → persists across reload → Clear reverts, the host Test button is live, and display mode renders the section — 17/17, zero uncaught page errors.
