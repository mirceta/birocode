Deploying the harness from inside the harness

How the 2026-07-06 23:54 deploy shipped main (d3384f2) to live :5099 — and why it needs a scheduled-task escape hatch and a dead-man switch.

Why you can't just run swap.ps1

This is Self-Development: the Repo being deployed is the Harness. The Claude session doing the deploy is a child process of the very server the deploy restarts.

🖥️ ClaudeWeb harness :5099 (live)
🤖 Claude session this conversation
📜 swap.ps1 foreground child
🗓️ Task Scheduler → deploy-keep-wrapper.ps1 outside the tree, RunLevel Highest

swap.ps1's job includes stopping the harness before swapping binaries…

The suicide problem

Foreground swap.ps1 stops the harness → the harness's process tree dies → the Claude session and the half-finished swap die with it. Live is left stopped.

The escape hatch

A scheduled task (ClaudeWebManualDeploy) runs the wrapper in a process tree outside the harness. The swap survives the restart; only the Claude session blips and resumes.

Why RunLevel Highest

The harness runs elevated. A non-elevated task can't kill its PID (this is exactly why the auto-rollback dead-man task was once inert — robocopy exit 11). The deploy task is registered RunLevel Highest so it can.

Today's wrinkle

Register-ScheduledTask was denied (0x80070005) — this shell isn't elevated. But the task persisted from the last deploy, so a plain schtasks /Run was enough. Starting a task you own doesn't need elevation; registering an elevated one does.

The dead-man switch

  
swap.ps1 snapshots live to run-bin.lastgood, then arms a 15-minute
  auto-rollback scheduled task. If nobody confirms the new build (keep.ps1), live silently
  restores itself to last-good. The wrapper only runs keep.ps1 after live answers
  200 — a broken deploy is never disarmed.

Probe loop

      
Up to 20 attempts × 2 s against /api/auth/check. One 200 → run
      keep.ps1 (disarm). Zero 200s → log “HEALTH FAILED → NOT disarming” and let the fuse burn.

    

    

      
Rollback = restore, not undo

      
rollback.ps1 restores with robocopy /MIR (never copy) from the
      run-bin.lastgood snapshot. And the timer uses a real DateTime trigger, never
      schtasks /SD — a locale bug bakes those two rules in.
