My recommendation: a new local app

Not a homepage topic. Hover either option — the platform supports both, but they pull in different directions.

🧪 New local app lab/

  • Audience = you, the researcher — your lab notebook.
  • Structured, growing, append-heavy data fits its own store.
  • Served like the Understanding app: /api/localview/{repo}/app/lab/.
  • Independent lifecycle — grows forever without touching onboarding docs.

🏠 Homepage topic

  • Homepage = onboarding doctrine for agents, not for you.
  • Static topic prose — would churn on every journal edit.
  • Mixes your evolving notes with frozen convention docs.
  • Only fits a tiny, rarely-touched set of notes.
Audience & purpose differ. doctrine for agents vs your own notebook.
Data model & cadence differ. structured growing records vs static prose.
Platform already supports it. multi-app per repo is shipped; Understanding app proves the pattern.
Independent lifecycle. journals grow; homepage topics freeze.

How it's served (build-less, like the Understanding app)

A static folder — lab/index.html + JS/CSS + vendored libs + data files — reached through the harness proxy. Relative URLs only, so a leading slash would escape the sub-path and 404.

Data shape — MVP (static, agent-curated)

Entries & patterns are JSON/Markdown in the repo that I add/edit on request: zero backend, full git history. A live browser-side CRUD store is an optional later slice.

An entry — lab/data/entries.json

{
  "id": "e-001",
  "kind": "learned", // found|testing|good|bad
  "title": "…",
