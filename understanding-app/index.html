Why this change exists

Local apps are embedded at their natural size. In the agent dock a full product is squeezed into a small tile; on the phone's Local tab some products render too big or too small for the screen.

Browser zoom can't help: it scales the whole harness — chat, tabs, composer — along with the app. What's needed is a zoom that touches only the embedded app's frame.

Wanted: per-frame zoom, on both embedding surfaces, harness chrome untouched.

What was decided this turn

  • New branch feat/local-app-zoom off an already-synced main (one feature per branch).
  • One new capability, local-app-zoom, covering both surfaces — no existing spec modified.
  • Purely client-side: no server or API work; the /api/localview/… proxy is untouched.
  • Basic-mode exception, on purpose: new UI normally defaults to Advanced, but the phone tab (End-User surface) was the explicitly requested target — so the Local-tab control is a Basic-visible viewing control.
  • Implemented & verified: 13/13 tasks done; a Playwright run against an isolated harness instance (own port + datadir, live untouched) passed all 23 checks — clamps, chrome unaffected, two-dock independence, reset-on-reload, composition with the dock zoom.

The load-bearing discovery: both surfaces share ProductFrame

Four places iframe a product, all through one shared component. The whole feature lands in ProductFrame.jsx behind an opt-in zoomable prop — the two target surfaces each change by one prop, and the two out-of-scope surfaces are untouched by construction (they never pass it).

Green — the two in-scope surfaces: dock (already Advanced-gated) and Local tab (zoom is Basic-visible, a viewing control).
Grey — App tab & Landing iframe the cross-origin preview port directly and stay exactly as they are.

Why state lives inside ProductFrame

Component-local useState gives the spec'd scope for free: each mounted frame has its own level (per-surface independence) and it's gone on unmount/reload (ephemeral, like the dock's maximize-chat state). The rejected alternative — host-owned state passed down — would duplicate control UI and plumbing in two hosts for no requirement we have.

Live demo — the exact CSS technique from the design

This demo is the mechanism: transform: scale(f) with the layout size compensated to calc(100% / f). Harness chrome around it never moves. (One mock-only difference: a plain div has no inner viewport, so here you scroll the dashed box; the real iframe fills its frame exactly and magnified content scrolls with the app's own scrollbars.)

Why not CSS zoom on the iframe?

      
    
        
  • zoom participates in layout; how it sizes an iframe's inner viewport differs across engines.
    • 
        
  • transform is paint-only and fully specified — identical at any nesting depth.
    • 
        
  • That includes inside the dock's already-zoom-ed screen, where the two simply compose visually (→ tab 4).
    • 
      

      
The two regimes

      
    
        
  • Always — layout 100%/f × paint scale(f) = exactly 100%: the scaled iframe fills the viewport at every level, so the frame's footprint never changes and there is no outer scrollbar.
    • 
        
  • f < 1 — the app sees a bigger inner viewport painted shrunk: more app at once, no dead margins.
    • 
        
  • f > 1 — the app sees a smaller inner viewport painted magnified; overflow scrolls with the app's own scrollbars, exactly like browser zoom (verified: inner width 499px → 250px at 200%).
    • 
      

    

  







  

    

      
Discovery: the dashboard already has a zoom — a different one

      
The Layout popover has a whole-dock content zoom (contentZoom in Dashboard.jsx): a persisted, per-device CSS zoom on every dock's .phone__screen — chat, console and all. The new frame zoom is per-frame, ephemeral, and composes on top; the spec pins both as independent.
