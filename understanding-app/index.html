The in-app GitHub PR browser

How the events-app GitHub tiles became a drill-down — repo → open PRs → one PR — with zero Chrome tabs and zero credentials in the browser. (openspec change github-pr-browser)

Why not an iframe? (the original ask)

github.com refuses to be embedded. Every github.com response carries X-Frame-Options: deny, so an <iframe> of a PR page renders a blank refused frame in every browser — no header, flag, or proxy trick makes it render without rewriting GitHub's HTML (fragile, against ToS). So the same outcome — no tabs — is delivered by rendering the PR data natively in the page, with an “open on GitHub ↗” escape hatch for deep review.

The contract, in five rules

✓Credential stays server-side. The browser calls the Harness; the Harness shells out to gh, which holds the PAT. No GitHub token ever reaches a phone.
✓Allow-listed repos only. ?repo= must be on the board's repo list — the repositories visible to the authenticated gh account (non-archived, most recently pushed first, capped at 100) unioned with the registered Repos' origin remotes (change github-repos-from-pat: every fleet machine pushes as the same account, so fleet repos appear without local registration). Anything else: 404 without contacting GitHub. The Harness can't be used as an open GraphQL proxy.
✓Escape-first rendering. A PR body is third-party content: it is HTML-escaped, then only http(s) URLs are autolinked. <script> in a description renders as visible text (Playwright-verified).
✓Panel-level degradation. gh missing / unauthenticated / timeout paints an inline ⚠ in the panel; the board, tiles, and page keep polling as if nothing happened.
✕Display mode stays inert. On the wallboard (?display=1) tiles have no click affordance and the panel never renders — glanceable, not a control surface.
