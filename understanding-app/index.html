The OpenSpec Cockpit

A proposed read-only tab for the Control Room that renders live OpenSpec state — what's in flight, what shipped, and what the system does today — the inspect-twin of the Console you already run commands in. This companion walks the ideas behind the proposal — and the one gap I had to close after building the drill-in.

The archive step is the one with no CLI behind it — openspec list reports active changes only — so shipped features are the gap the old plans/* board never surfaced cleanly. The Cockpit closes it by reading openspec/changes/archive/ directly.

Two tabs, one posture: drive vs read

The Console already drives OpenSpec through a whitelisted POST ./api/exec. The Cockpit is its mirror — it only reads. Nothing that mutates an artifact ever moves out of the Console.

🛠️Console mutates

exists today — how you operate OpenSpec

▶Propose / archive / validate — runs the verbs
🔒strict whitelist: server builds the argv, client picks verb + safe id
📄shows raw text output — great for learning the CLI

🛰️Cockpit read-only

proposed — how you read OpenSpec at a glance

👁️In flight · Shipped · Baseline — rendered as cards, not text
🔌one read endpoint GET ./api/cockpit; drill-in via ./api/cockpit/show
🚫no new write verb joins the whitelist — security story unchanged

Why read-only is a hard line: the Cockpit can never half-run a destructive verb, and the only new server surface reads state. It shows the spec world; CI is still what enforces it.

The ring knew 14 / 18 — but the checklist was invisible

The "In flight" card shows a completion ring, and clicking it drills in. The ring was right; the drill-in showed deltas but no task list. Two different commands feed those two things — and only one of them carries the tasks.

🟢The ring

openspec list --json
per change: { … completedTasks: 14, totalTasks: 18, status }
✓ carries the count — so the ring drew 14/18 (78%) correctly.

🔴The drill-in

openspec show <id> --json
{ id, title, deltaCount, deltas }
— no tasks key at all —
✗ has the deltas, but nothing to render the checklist from.

So the Cockpit could tell you "14 of 18 done" without being able to show you which 14 — exactly the gap you spotted. The count and the list came from different places, and the drill-in only had the half without the list.

The fix: read the same file the ring counts

The task definitions live in one place — the change's tasks.md, a flat list of - [ ] / - [x] checkboxes. That's the very file list --json counts. So the server now reads it directly (the same move it already uses for the archive folder, which also has no CLI) and attaches it to the drill-in.

📖
Server — serve.mjs gained a readTasks(id): open openspec/changes/<id>/tasks.md, parse each checkbox into { done, text, section } grouped by its ## heading, and attach it as tasks on the ./api/cockpit/show response — only for changes, never for specs.
🎨
Client — app.js renders that list as a read-only checklist below the deltas: a done/total header, section sub-headers, and a ✓ box per item. (Bonus: requirement & task text now renders inline `code` and **bold** instead of literal markdown.)
📐
Spec — the delta requirement said the checklist came from show --json, which isn't true. Corrected to state it's read from tasks.md; re-validated --strict → valid.
