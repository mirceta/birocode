OpenSpec change · proposal

add-cockpit-openspec-setup

Make the OpenSpec Cockpit able to port a repo into OpenSpec with one click — turning its read-only readiness banner into an action — while deliberately and explicitly relaxing the cockpit's read-only guarantee through a spec delta, not by sneaking past it.

The problem, in one picture

today

The cockpit detects a repo isn't OpenSpec-ready and prints static text:

Run openspec init in this repo, then reload.

You leave the harness, open a shell in that repo, run the CLI by hand, come back, reload.

→
this change

The same banner offers a button:

Runs openspec init --tools claude in the selected repo, re-checks readiness, refreshes in place. No shell, no reload.

Why now

You want OpenSpec in other repos too (e.g. prg-copy1). The standalone Control Room already documents this as Operate → Workflows → "Set up the tool" (openspec init --tools claude, then openspec update). This brings that one capability into the cockpit you already use.

The key insight: repo scoping

The cockpit is already scoped to the selected repo via the X-Repo-Id header → RepositoryResolver → working dir. So you port prg-copy1 by selecting it in the repo picker — not by hardcoding a path. "Set up this repo" needs no path parameter at all.

⚠ A convention conflict — surfaced, not buried

Per the repo's most important convention (CLAUDE.md), a request that breaks a written convention must be flagged before acting. This one does:

openspec/specs/openspec-cockpit/spec.md — "read-only … no mutating verb"

A Set up OpenSpec button runs openspec init — a write. That directly contradicts the cockpit's read-only baseline.

The resolution is not to ignore it, but to MODIFY the requirement through the change's delta spec — the legitimate OpenSpec mechanism — carving out exactly one explicit, gated exception:

  • Fixed verbs only — init --tools claude or update; the API never accepts a command string or arbitrary args.
  • No-clobber guard — init refuses to run if openspec/ already exists (enforced server-side, unreachable from the UI).
  • Repo-scoped — runs only in the resolver-resolved working dir; can't target another directory.
  • Everything else stays read-only — in-flight, shipped, baseline, drill-in are untouched.

If you'd rather keep the cockpit strictly read-only and build this in the standalone Control Room, that's the alternative — say so and it redirects before any code.

MODIFIED A harness-native read-only Cockpit tab…

The read-only clause is rewritten to name the single exception:

− “The tab SHALL NOT expose any action that … mutates OpenSpec artifacts — it is read-only.”
+ “The only state-changing action the tab MAY expose is the explicit, gated setup action; no other mutating verb SHALL be exposed.”

And the "No mutating verb" scenario becomes "No mutating verb beyond the gated setup action".

ADDED Set up a repository for OpenSpec from the Cockpit

One endpoint, fixed verb, no-clobber guard, repo-scoped, readiness re-run after. Six scenarios:

Set up a not-ready repo — WHEN CLI present & no openspec/ THEN run init, re-check, reflect ready without reload
Refuse to clobber — WHEN openspec/ already exists THEN don't run init; report "already initialized"; tree untouched
Unavailable without CLI — WHEN no CLI on PATH THEN don't offer init; direct to install
Refresh instruction files — WHEN already initialized THEN run openspec update, surface result
Fixed verb & scope enforced — WHEN endpoint called THEN only the fixed verb in the resolved dir; reject arbitrary args
Surface failure cleanly — WHEN verb exits non-zero THEN show captured result as failure; readiness reflects true state

One new state-changing endpoint

POST /api/openspec/setup { "action": "init" | "update" } // a discriminator — never a command string

Maps to a fixed verb server-side, runs it in the resolved repo working dir via the existing RunOpenspec(workingDir, …), then returns command result + refreshed readiness.

Controller — OpenspecController.cs

  • Resolve dir via RepositoryResolver (same X-Repo-Id path as reads)
  • Validate action ∈ {init, update}
  • Return { ok, action, exitCode, stdout, stderr, alreadyInitialized?, ready }

Service — OpenspecCockpitService.cs

  • RunSetup(workingDir, action): init→init --tools claude, update→update
  • init: if openspecDirPresent → "already initialized", run nothing
  • After run: CheckReadiness() again, return both

Frontend — Cockpit.jsx readiness section

No new tab/section; the existing "Prepared for OpenSpec?" block becomes conditional on data it already has:

  • Set up OpenSpec when openspecOnPath && !openspecDirPresent
  • Update instruction files when openspecDirPresent
  • install-CLI hint when !openspecOnPath (init can't run without it)
  • after POST → re-run the existing load() so the whole tab refreshes in place

Two decisions left for you

a. Should a successful init auto-chain openspec update, or stay two separate buttons?

Leaning: separate, matching the Control Room's two-step workflow.

b. Should the result show the scaffolded file tree, or just success + refreshed readiness?

Leaning: just readiness for this change; richer output can follow.
