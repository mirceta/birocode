The agent-dock Event Console

The harness keeps moving work to be backend-owned — discovery scans that survive a refresh, apps it launches, ports it probes, autopilot and loops that drive a repo on their own. Each is something the harness does to a repository, yet the dock gives no window into it. This change adds that window: a per-repo Event Console lane that narrates, at the boundary our code controls, that an operation was invoked and we're awaiting it — then that it returned and what the harness did with the result. Deliberately shallow: the agent gateway stays a black box.

The one rule — log only the boundary we own

Every event is something our application did or saw at the edge of an operation. We never reach inside the ClaudeMonitor gateway to report which tools a sub-agent called or its token usage — that's the gateway's own domain and a much larger change. Shallow first; deepen only if a real need appears.

Try it — invoke operations, watch the Console fill

Click the operations. Each emits a started event into the backend's per-repo ring, then a terminal event when it returns. The dock's Console lane polls ?after=N on a timer and renders new events chronologically — so events appear a poll-tick after they're emitted, exactly like the real ~5s dock cadence (sped up here).

What you're watching

  • Emit at the boundary. Each op pushes a small record { seq, at, op, phase, title, detail } into the repo's ring. started first; a done/error when our code sees the result.
  • Monotonic seq, capped ring. Seq only ever increases; past the cap the oldest events drop (greyed) — same discipline as the chat run buffer. In-memory only.
  • Poll by watermark. The lane holds an after seq, asks for seq > after, then advances it by the returned lastSeq — never re-receiving what it has seen.
  • Per-repo, not per-tab. Two docks on one repo read the same log — that's the truth of where jobs are keyed.

Where each piece lives

Backend (ClaudeWeb.App)

  • RepoEventLog — singleton; ConcurrentDictionary<repoId, RepoLog>, each a locked list + monotonic _seq + soft cap.
  • EventRecord { Seq, At, Op, Phase, Title, Detail } — Op/Phase are open strings, so autopilot/loops can emit later with no model change.
  • Emit calls in LocalAppDiscoveryJobs (start/done/error) and the run/check endpoints.
  • GET /api/repos/{repoId}/events?after=N → { events, lastSeq }; normal session auth.

Frontend (client/)

  • New Console lane button in PinnedAgent.jsx, beside Builder / Ask / Files.
  • EventConsole — scoped by repoId; polls at the dock cadence, advances its watermark, renders chronologically.
  • Capability key 'advanced' in UiModeContext.jsx; i18n + dashboard.css.

Key design decisions

Slice 1 (this change)

  • One source: the three discovery-feature ops we just built — Discover, Run, Check.
  • Lifecycle depth only — invoked / awaiting / returned + what we did. No gateway internals.
  • Polling at the existing dock cadence; in-memory ring; per-repo scope; defaults Advanced.

Out of scope (a later slice, only if needed)

  • Tool-by-tool gateway detail (needs tool-use events threaded out of ClaudeMonitor — bigger).
  • Autopilot-intercept & loop-engine sources feeding the same console.
  • SSE transport · cross-restart persistence.

Decisions & trade-offs

Per-repo or per-tab?
Per-repo. Jobs are already keyed per repo (discovery is per-repo single-flight); two docks on one repo see the same activity. Per-tab would need attribution plumbing the backend doesn't have.
Reuse RunSession?
Borrow the shape, not the machinery. Seq + watermark + cap, yes; the live Channel<> push, no — append-only + poll is enough for slice 1.
Transport
Poll ?after=N, not SSE. Matches the discovery-status poll already in the dock; SSE is an additive upgrade if 5s feels laggy.
Persistence
In-memory ring, capped. A restart means "nothing happening now" — acceptable for a live console. Disk persistence is out of scope.
Event depth
Boundary only. later tool-level. Honest wording: "returned N apps", not a UI action the backend can't observe; "launch issued", not "running" (no PID retained).
Model shape
Open strings for Op/Phase. Source-agnostic so autopilot/loops emit later without touching the model or the renderer.
The one rule that explains the whole thing:
  • Log the boundary, not the black box. We record that the harness invoked something, is awaiting it, and what it did with the result — never what the agent gateway does inside.
  • The log is per-repo and append-only, with monotonic seq and a cap; the dock reads it by watermark, the same reattach contract chat and discovery-status already use.
  • Built source-agnostic and shallow — discovery/run/check today, autopilot/loops a one-liner away; deepen to tool-level only if a real need appears.
