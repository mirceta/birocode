Why not a React-client tab or a standalone repo? The events-app pattern is proven for exactly this shape — always-on, no build step, iterate by overwriting a folder — and the harness already knows how to serve it. A standalone product would drag in a second server for what is fundamentally a harness view.

This isn't invented — "mission control" boards are an established practice in agentic engineering. Mission Control, claude-view, Conductor, AgentsRoom, and Marc Nuri's cross-device dashboard (his problem is literally ours: 5–10 sessions across machines) all converge on the same shape.

The consensus board
  • per-machine / per-agent cards
  • status: working / idle / awaiting input
  • current task, project + branch, PR link
  • context-window %
The two consensus insights
  • awaiting-input outranks everything — the human is the bottleneck, so the board's real job is the attention queue of "things blocked on me"
  • context % is the most actionable per-agent metric (ours once the feed carries it)
Companion panels people run
  • cross-repo GitHub wallboard: PRs awaiting review, red/green CI (GitactionBoard, GitHub repo dashboard) — in v1
  • usage/burn-rate ticker (ccusage, Claude-Code-Usage-Monitor) — deferred, needs per-machine JSONL
Wallboard rules (Hanselman's "third monitor")
  • it's the "everything changing" screen
  • glanceable from across the desk, zero interaction
  • alert-first ordering
  • must visibly go stale — a frozen board that looks healthy is worse than no board
The birocode punchline: everyone else's dashboard has to invent its data plane. Ours already exists — the event-feed collector polls the other machines' harnesses today. This change is mostly a view over it.
In v1
  • new capability status-monitor (spec'd, strict-valid)
  • status-app/ build-less SPA + StatusApp service (mirrors events-app)
  • GET api/status-monitor/board — fleet / attention / github in one doc
  • attention queue v1 = refusal-state sources + stale sources (source-level)
  • GitHub panel: explicit repo list, PRs + review state + default-branch CI, ≥60s cache
  • read-only; wallboard presentation is a requirement, not styling taste
Deliberately deferred (each its own follow-up change)
  • feed enrichment: per-agent awaiting-input, current task, context % (modifies harness-event-feed + event-feed-collector — kept out so this change stays additive)
  • usage / burn-rate panel
  • acting from the board (approve / dismiss / retry)
Open questions (design.md)
Which repos on the GitHub panel?
settings list vs derived from registered Repos' remotes — lean: start with a settings list.
Staleness source?
does the collector already persist last-successful-poll per source, or does the board compute it from events? Audit is task 1.1.
Sound on new attention items?
events-app has a sound endpoint — lean: silent v1, the monitor is peripheral vision.
