Decision · feature/openspec-flow

Reconcile our system to OpenSpec — or take one idea out of it?

The real fork in the road. Two paths, what each truly costs, and — since you asked what I think — a clear recommendation with the reasoning behind it.

Same goal, two very different bets

Both close the gap you now see. They differ in how much they move, what they risk, and how hard they are to undo. Tap the tabs on each card.

Path A

Reconcile our system → to OpenSpec

Adopt the tool: its specs//changes//archive/ dirs, its CLI and /opsx flow. Our plans become OpenSpec changes.

  • Restructure plans/* into OpenSpec's folders + naming.
  • Install + learn a second toolchain; agents drive it via /opsx.
  • Author proposals/designs/tasks/deltas in its layout.
  • A proven, maintained flow with 25+ assistant integrations.
  • Strict, uniform structure + GIVEN/WHEN/THEN scenarios.
  • The living baseline comes built-in.
  • Two sources of truth — its dirs vs our plans + harness — unless we rebuild harness rendering around it (big project).
  • Churns existing plans + history for things we already have.
  • Risks our real moat: harness integration.
  • Ceremony (scenarios everywhere) we mostly don't need.
Path B

Take one idea → into our system

Lift only what we lack — the living baseline + change-as-delta — into our existing convention. No new tool, no new dirs.

  • Add one living doc (docs/capabilities.md = what it does today).
  • Add a one-line delta stanza to each plan (ADDED/CHANGED/REMOVED).
  • Add one ritual step: on ship, fold the delta into the baseline.
  • Closes the only real gap — queryable "what does it do today?".
  • Stays one source of truth, rendered in the harness we already have.
  • Cheap, and trivially reversible if it doesn't earn its keep.
  • We build the small mechanism ourselves (a doc + a habit).
  • No CLI / slash-commands / ecosystem.
  • Discipline rests on our ritual, not a tool gate.

The two facts that decide it

Strip away the detail and the choice rests on just two things.

Fact 1 — How much of OpenSpec is actually new to us?

We already do most of spec-driven development. Of OpenSpec's parts, only one is net-new — so "adopt the whole tool" buys mostly things we have.

  • proposal ≈ understanding.mdhave
  • design ≈ planhave
  • changes/ ≈ branch + planhave
  • archive ≈ recently-shippedhave
  • CLI/opsx ≈ harness renderinghave
  • tasks.md (tighter checklist)partial
  • specs/ baseline + deltasNEW

Buying a whole toolchain to obtain one idea is a bad trade — when the idea is separable. And this one is: the baseline needs no CLI and no dirs.

Fact 2 — Where does the source of truth live?

Harnessplans + capabilities.md, rendered live
OpenSpec dirsspecs / changes / archive

What I'd do

Path B

Take the one idea — reconcile it into our system. Don't reconcile our system to the tool.

We already do ~90% of spec-driven development, wired into the harness. OpenSpec's value to us is concentrated in a single thing we lack — a living baseline. Lift that; leave the rest.

The reasoning

  1. The value is separable. The baseline + delta works as a plain doc + a ritual step. It does not need the CLI or the dirs — so we don't have to swallow the tool to get it.
  2. Don't trade away the moat. Our edge is harness integration (Plan tab, doc viewer, Understanding panel, deploy ritual). Path A either loses it or forces a big rebuild to keep it.
  3. One source of truth. Two stores is the exact drift we keep fighting. Path B keeps everything in the harness; Path A re-introduces the split.
  4. Cheap and reversible. Path B is an afternoon and undoable; Path A churns history and is hard to walk back.
One honest caveat — calibrate first. Before committing, run one throwaway propose → apply → archive cycle with the real OpenSpec. A flow benefit can hide until you feel it (its tasks.md rigor is the likeliest). If calibration surfaces another worth-having habit, borrow that too — still ideas into our system, not our system into the tool.
  • If we had no existing convention to build on.
  • If matching the wider tool ecosystem (25+ assistants) were a hard requirement.
  • If we wanted the strict GIVEN/WHEN/THEN contract style everywhere, not just where a verifier exists.

None of these hold for us today — which is why the recommendation is B.
