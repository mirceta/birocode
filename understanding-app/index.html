Understanding · how the port actually lands

The OpenSpec transition — flip on a branch, catch the stragglers

Solo repo, but agents are working other features in parallel. The plan: do the whole OpenSpec port on feature/openspec-flow, merge it to main only when it's complete, and let CLAUDE.md + a migration recipe turn every straggler branch's merge-conflict into a guided translation — so nobody is left wondering what broke.

1Transition on the branch Phases 0–3 land on feature/openspec-flow. plans/* stays alive there until it's done. Nothing on main changes yet.
→
2Merge → the convention flips CLAUDE.md's interim disclaimer is swapped for the OpenSpec block; plans/* freezes; openspec list becomes the dashboard.
→
3Stragglers translate & merge A branch still on plans/* merges main, hits a conflict, reads the new convention, and follows the recipe. Walk it below.

Three artifacts, three jobs

CLAUDE.md

The signal. Injected into every agent. Today: "in transition — keep using plans/*." At merge it flips to the OpenSpec block (parked in plans/openspec-flow.md) that routes stragglers to the recipe.

docs/openspec-migration.md

The recipe (source of truth). Agent-agnostic, read off disk: old→new map + the exact steps to convert a plans/* feature into an OpenSpec change and resolve the merge.

openspec-port-app · Migrate tab

The operate surface. Type the feature name, copy a finished prompt that walks an agent through the recipe end-to-end. Operationalizes the doc; doesn't restate it.

Companion to plans/openspec-flow.md (Phase 4) and docs/openspec-migration.md · the transition is staged, not yet merged.
