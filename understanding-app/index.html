Agent audit trail openspec: add-agent-audit-trail

Two dashboard buttons launch autonomous agents — Discover local apps 🛰️ and Ask for understanding 🧠 — and today the harness keeps no durable record of who ran them. This change adds an append-only audit of every agentic call, plus a read-only trail view in the web UI.

The gap

Anyone with phone access can press a button that spawns an agent reading a repo (discovery) or writing to it (understanding). After a restart, nothing answers: who ran an agent, when, on which repo, and how did it end?

Today — RepoEventLog only

  • In-memory, capped at 500 events, gone on restart
  • Built for live status streaming, not accountability
  • Job registries keep only the latest run per repo — history overwritten
  • No actor, no IP — can't tell operator from phone guest

After — AgenticAuditLog

  • Durable append-only JSONL: agentic-audit.jsonl in app data
  • Every call: timestamp, feature, repo, actor + IP, outcome, duration
  • Read-only trail view on the dashboard (Advanced mode)
  • Generic emit surface — the next agentic feature registers with two calls

What is deliberately NOT in scope

Metadata only

No prompt text, no tool calls, no agent output. That restraint is exactly what makes a web-visible trail compatible with the existing action-audit stance (which is desktop-only). See view 4.

Not chat, not Autopilot

Chat lanes and Autopilot already have their own records (audit/*.jsonl, autopilot-audit.jsonl). This trail covers the button-triggered agentic features only.

No retention/rotation v1

Volume is human-button-press scale. If it ever matters, the daily-rotation pattern from AuditService is a drop-in — store-internal change, spec unchanged.

One call, end to end — press the buttons

The controller resolves who (request-scoped); the job registry owns lifecycle (start vs join, terminal state even if the phone disconnects). Two entries per call, correlated by callId.

Append-only store → merged call rows

An append-only file can't update a row, so each call is two entries sharing a callId. The read endpoint folds them into one call row; the pair is also what makes crashes visible instead of silent.

State rules

done / error / canceled — a terminal entry exists for the callId. Errors carry a trimmed summary, never the agent transcript.
running — started with no terminal and the job registry has a live job for that repo. The dashboard panel polls every 5s while any call is in this state.
interrupted — started with no terminal and no live job (the process died mid-run). This is why the start entry is written eagerly: writing only at completion would make crashed runs vanish entirely. Never renders a phantom "running" forever.

Why emit from the registries, not the controllers?

Registry knows the lifecycle

Only StartOrJoin knows whether a request actually started a job or joined one — and the terminal fires inside the job task regardless of clients.

Controller knows the identity

Actor resolution needs the HttpContext. So the controller resolves actor + IP and passes them into StartOrJoin. Identity is request-scoped; lifecycle is job-scoped.

Two audit stores, one deliberate boundary

The existing action-audit spec says its store is desktop-only — the web must never read it back. This change does not bend that rule; it creates a second, physically separate store whose content is safe to show on a phone.

Why not extend AuditService instead?

Mixing both datasets in one store would make the new web endpoint a spec violation waiting to happen — one filtering bug away from exposing prompt text to the phone. A separate file makes the boundary physical, not a query condition.

Guarantees on the new store

Append-only

Entries are never rewritten or truncated; restarts leave history intact.

No mutation from the web

The controller has no mutating verbs at all — there is nothing to lock down because nothing exists.

Advanced-gated view

New capability agenticAudit: 'advanced' in UiModeContext.jsx — Basic mode never sees it (per the UI-modes convention).
