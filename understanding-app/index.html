Understanding · do Control and Console overlap?

Control vs Console — same thing, or two axes?

You spotted real overlap: Control’s Phase 0 and Console’s “Set up the tool” both cover getting OpenSpec installed. But they sit on different axes. Control tracks the one-time port project — five phases with an end (Phase 4 is the point of no return). Console runs the recurring jobs you do forever after. They’re the track ↔ run pair, not duplicates. Click any node to see its counterpart.

Only three rows line up. Phase 3–4 are project-only (nothing to run); Inspect & Hand off are job-only (no phase). The overlap is just where a phase happens to be a job.

Zoom: Phase 0  vs  “Set up the tool”

Even where they overlap, they’re not the same. Phase 0 is a 5-step milestone checklist; only one step is an OpenSpec command. Console’s set-up is the executable subset — plus two tooling pings Phase 0 doesn’t track.

So — merge them?

No

Don’t merge the tabs. One is a phased project that finishes; the other is the daily driver that never does. Folding them would conflate “port the system” with “use OpenSpec” — different lifespans, different readers.

But do this

Wire run → track. When you click init in Console, auto-tick Phase 0’s “Run openspec init” in Control, and show Phase 0: n/5 in the Console set-up header. One source of truth for state — that’s the “more under control” you’re after.

Honest caveat: the run→track link is clean for set-up (init → one task), fuzzier elsewhere — validate --strict touches both Phase 1 and Phase 2, so not every button maps to exactly one checkbox.
