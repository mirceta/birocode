Interactive + headless system tests

One test definition, two ways to run it. An agent runs it headless and gets a verdict. An operator steps through it and watches each step pass or fail. Toggle the mode below and drive the mock runner.

    Headless: every step fires back-to-back, you just read the verdict.

    What makes a test steppable

    The unit is a step. Same check() calls as today — just wrapped so each scenario has a boundary the harness can pause on and report.

    // lib.mjs — the new boundary
await step('Auth gate rejects no-cookie', async () => {
  const r = await api('/api/chat', {noAuth:true});
  check('POST /api/chat → 401', r.status===401);
});

// headless  → steps run back-to-back
// interactive → blocks here until the
// operator clicks "Next step"
    // structured events the hub reads
{type:"step",phase:"start",i:0,name:"Auth gate…"}
{type:"step",phase:"end",i:0,
   status:"pass",
   observed:"401 as expected",
   checks:[{name:"… → 401",pass:true}]}
// + the old [PASS]/[FAIL] console lines stay,
// so nothing that reads them today breaks.

    How a click becomes a step (interactive)

    SPA

    Operator clicks Next step.

    hub/public
    →
    Hub server

    Writes go to the live child's stdin.

    server.mjs
    →
    Runner

    Unblocks, runs one step(), emits start/end events.

    lib.mjs
    →
    SPA

    Step card lights up pass/fail with its observed detail.

    SSE

    Headless skips the wait entirely — the runner never blocks, the child just runs to exit.

    Why one definition, not two

    ✓ Single source

    The same .mjs drives both modes. Add a step, fix an assertion — both the agent's headless run and the operator's click-through get it. Nothing drifts.

    ✗ Two copies (rejected)

    A separate "interactive" script would slowly diverge from the headless one until they test different things — the classic test-harness rot. Avoided.
