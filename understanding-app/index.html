What pressing it does

  • Grabs this dock’s conversation (its builder sessionId).
  • Forks it into Claude Monitor — your live chat is never touched.
  • The forked agent writes understanding-app/index.html explaining your latest turn.
  • Progress streams into the same Console lane Discover already uses.
It’s exactly the “clarify that in the Understanding app, with demos & diagrams” follow-up you do by hand — now on one button.

It stands on Discover’s shoulders

Discover proved the pattern: a dock button → a backend-owned Claude Monitor job → Console events. This change mirrors that shape so it inherits disconnect-resilience and shows up in the same Console — nothing parallel is invented.

Why snapshot-resume (fork) — not continue (resume in place)

Your dock conversation is your live chat. Resuming it in place would interleave a long file-writing turn with whatever you do next. Snapshot-resume copies the transcript to a fresh session and runs there — capturing the conversation “at that time,” exactly as you asked.

The “more advanced” part = it writes files

We checked: Claude Monitor already allows writes. It just spawns the claude CLI. Discover is read-only only because our harness pins --allowed-tools Read Grep Glob LS. The escalation here is ours, not a Monitor limit.

Option A — accept the CLI default toolset

The resume command is literally:

claude -p - --resume "<id>" --output-format stream-json --verbose

No --allowed-tools. And SnapshotResumeRequest has no AllowedTools field to add one. So the fork runs with the CLI’s default permissions — which include Write/Edit.

Chosen: accept the default. No Claude Monitor change. The fork can write the Understanding app today.

Trade-off we accept: we can’t drop Bash or otherwise bound the toolset without adding an AllowedTools knob to snapshot-resume (that was the rejected Option B).

What bounds the blast radius instead

  • Working dir = the repo root → writes stay in this repo.
  • Prompt scopes the task to understanding-app/ — a folder the convention says to overwrite anyway.

Backend-owned, latest-only, observable

Modeled on LocalAppDiscoveryJobs so a phone refresh mid-run never cancels it.

  • Own cancellation token — survives request abort / disconnect.
  • Start-or-join, latest-only — a second press joins the running job; a press after a finished one replaces it.
  • Two endpoints: POST /api/understanding/ask (start-or-join) and GET /api/understanding/status (reattach only — never starts).
  • Console events op="understanding" · phases started / done / error. EventConsole.jsx already renders any op — no Console UI change.
