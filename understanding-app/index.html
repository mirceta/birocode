Two deploy dead-man's switches, side by side

The merge of main into feature/deploy-deadman-switch left two parallel implementations of the same safety feature in the tree. They don't share code and partly collide. This walks both, shows where they clash, and what a reconciled design looks like.

The one-sentence version

Your branch (#2) is the working safety loop with no UI; main (#3) is the working UI shell with a hollow safety loop. Neither is a superset of the other — so "is the dead-man switch a sure improvement?" is no, not as-is: it's better than the bare portable swap it replaced, but it now sits next to a more complete observability layer it doesn't talk to.

Reproducible provisioning, a deploys.jsonl ledger, and an operator UI (live state, "Keep it", "Trigger rollback"). But its swap never snapshots and never arms the timer — see the Main flow tab.

The four concrete breakages if shipped as-is

🟥
UI goes blind. Branch swap.ps1 never writes deploys.jsonl, so main's Deployments tab shows empty live/history panels.
🟥
"Trigger rollback" hits the wrong script. DeployService.TriggerRollback() runs the off-repo rollback.ps1, which looks for a .lastgood that only the in-repo branch swap creates — under a different path. Nothing to restore.
🟧
Two deploy targets. Branch deploys Debug run-from-copy; main's template mirrors into bin\Release in-place — the exact locked-bin model plans/portable-deploy.md calls the wrong assumption.
🟨
One lucky alignment. Both use task name ClaudeWebAutoRollback, so the UI's "Keep it" would disarm the branch's armed timer. Coincidence, not integration.

Completeness, scored

Branch #2 — the safety loop works
  • ✅ snapshot → arm → keep/rollback is a closed loop
  • ✅ real DateTime trigger (no /SD locale bug)
  • ✅ /MIR everywhere, runtime state protected
  • ❌ no ledger, no operator visibility
Main #3 — the shell works, the loop is hollow
  • ✅ reproducible provisioning from templates
  • ✅ ledger + Deployments tab + keep/rollback buttons
  • ❌ swap never snapshots → rollback has nothing to restore
  • ❌ swap never arms → the timer the UI manages is never set

Recommendation: fold the branch's mechanics into main's shell

The shell (provisioner + ledger + UI) is the harder half to recreate and it's already on main. The branch owns the harder-won half of the mechanics. Move the mechanics into the template, not the reverse.

branch: snapshot + arm + run-from-copy ⟶ into ⟶ main: templates + ledger + Deployments tab = one complete system

Target deploy flow (merged)

Concrete steps

  • Port snapshot-before-swap + timed arm (call arm.ps1) into swap.ps1.tmpl — the template's swap is currently missing both.
  • Make the swap write deploys.jsonl (main's template already shows the one-line shape) so the UI lights up.
  • Pick one target: adopt the branch's run-bin run-from-copy (safer than locked in-place Release) and align the .lastgood path so the UI's rollback button restores the right snapshot.
  • Pick one location: off-repo (main's rationale — scripts survive a working-tree revert) and delete the redundant root scripts so there's a single source.
  • Do it as an OpenSpec change (/opsx propose) — this is cross-cutting deploy safety, exactly what spec-driven review is for.
